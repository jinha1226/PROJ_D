# PocketCrawl — Skill System & God/Essence System Design Discussion

## 프로젝트 개요

PocketCrawl은 DCSS(Dungeon Crawl Stone Soup)의 모바일화를 목표로 하는 로그라이크 게임.
GDScript 4 / Godot 4 기반. 터치 UI에 맞게 시스템을 단순화하되 DCSS의 핵심 재미를 유지하는 것이 목표.

---

## 1. 스킬 시스템

### 현재 구현 (5개)

```
melee    — 근접 무기 전체
ranged   — 활, 석궁, 투척 무기 (현재 모두 묶여있음)
magic    — 주문 전체 + 지팡이
defense  — 갑옷, 방패
agility  — 회피, 은신, 기습 데미지
```

### 논의 중인 변경안 (6개)

```
melee    — 근접 무기 전체 (유지)
ranged   — 활, 석궁 전용 (좁힘)
magic    — 주문 전체 + 지팡이 (유지)
defense  — 갑옷, 방패 (유지)
agility  — 회피, 은신, 기습 데미지 (유지)
tool     — 완드 + 투척 무기 (신규)
```

### 변경 이유

- DCSS에서 Evocations(완드/마법 아이템)와 Throwing은 별도 스킬
- 현재 `ranged`가 활과 투척을 모두 묶어 플레이스타일 구분이 모호
- `tool` 분리 시 각 클래스 정체성 명확해짐:

| 클래스 | 주력 스킬 |
|--------|-----------|
| Warrior | melee + defense |
| Ranger | ranged + agility |
| Mage | magic |
| Rogue | agility + tool (투척/완드) + melee(단검) |
| Archmage | magic (전 학파) |

### Rogue 클래스 포지션

- 기존: ranged + agility → ranged 스킬이 활과 겹쳐 정체성 모호
- 변경 후: agility(기습 데미지) + tool(투척) + melee(단검) 소량
- DCSS Assassin과 동일한 구조: 기습 첫 타가 강력 → melee 스킬 낮아도 실용적
- agility에 기습 데미지 보너스가 포함되어 있으므로 3스킬 분산이 허용 가능

### 완드의 귀속 문제

- 완드를 magic에 넣으면: 비마법 직업(Warrior 등)이 완드 사용 시 어색
- 완드를 tool에 넣으면: 모든 직업이 자연스럽게 사용 가능, Mage는 MP 보조로 활용
- **결론: tool에 귀속**

## 2. 에센스 시스템

### 현재 구현

- 몬스터를 처치하고 룬을 소모해 에센스 장착
- 에센스 슬롯: body / unique (각 1개)
- 패시브 효과 + 액티브 효과 부여
- 예시: `essence_undeath` = drain_touch 패시브 + resist_necro 효과

### 에센스 목록 (현재)

```
essence_plague    — 독 공격 강화, 독 피해 +20%
essence_tempest   — 원거리 공격 +15%, 전기 저항
essence_stone     — AC +4, 이동 속도 감소
essence_gloam     — 비인지 대상 첫 타 +35%
essence_undeath   — drain_touch 패시브, necro+ 저항
```

### 에센스 시스템의 역할

- 플레이어가 런 중에 발견한 몬스터에서 능력을 흡수
- 빌드 커스터마이징의 한 축
- "어떤 에센스를 모을까"가 중기 목표가 됨

## 3. 신 시스템 (미구현, 논의 중)

### DCSS 원본

- 신 수: 20개 이상
- 경건도(piety) 0~200, 6티어
- 각 신마다 고유 규칙 (Trog: 마법 금지 / Cheibriados: 이동 속도 제한 등)
- 신 능력 3~5개, 신 변절 시 패널티

### 포켓크롤 단순화안

- 신 수: 6~8개
- 경건도 티어: 3단계
- 능력: 티어당 1~2개
- 후보 신:

| 신 | 성향 | 핵심 기믹 |
|----|------|-----------|
| Okawaru | 전투 | 무기/방어구 선물, 전투 강화 |
| Trog | 야만 | 마법 금지, 근접 강화 |
| Makhleb | 혼돈 | 킬 시 HP 회복, 악마 소환 |
| Vehumet | 마법 | 마나 회복, 고레벨 주문 지원 |
| Yredelemnul | 죽음 | 언데드 소환, 드레인 강화 |
| Cheibriados | 시간 | 느리게 이동 시 강력한 보너스 |

### 에센스 vs 신 시스템 비교

| 항목 | 에센스 시스템 | 신 시스템 |
|------|--------------|-----------|
| 구현 복잡도 | 낮음 | 높음 |
| 선택 시점 | 런 중 점진적 | 초반 1회 선택 |
| 런 방향성 결정력 | 중간 | 매우 높음 |
| DCSS 정체성 | 낮음 (독자 시스템) | 높음 (핵심 기믹) |
| 리플레이성 기여 | 중간 | 높음 |
| 빌드 강제성 | 약함 | 강함 (Trog 마법 금지 등) |
| 모바일 UI 부담 | 낮음 | 중간 |

### 핵심 질문

1. 에센스와 신이 둘 다 "특수 능력 획득" 역할을 하는데 **역할을 어떻게 분리할 것인가?**
   - 안 A: 에센스 = 몬스터 킬 기반 패시브 강화 / 신 = 플레이스타일 방향성 결정
   - 안 B: 신 시스템으로 통합 (에센스 제거)
   - 안 C: 에센스 유지, 신은 추후 추가 (현재 미구현 유지)

2. Trog처럼 **제약을 주는 신**이 모바일 환경에서 재미있을까, 불편할까?

3. 신 능력 발동을 **능동(MP처럼 소모)** 으로 할지 **수동(piety 기반 자동)** 으로 할지?
